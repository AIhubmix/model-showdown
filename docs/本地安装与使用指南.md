# Model Showdown 模型对比视频工具 — 本地安装与使用指南

> 每周 1-2 条「多模型同题对比」短视频的本地生产流水线。
> 一句话流程：**同一 prompt one-shot 发给多个模型 → 自动录制成品运行画面 → 合成三面板对比视频（英文 + AIHubMix logo）→ 产出 X/Reddit/HN 发布素材包。**
> 代码位置：`~/code/model-showdown/`（2026-07-17 建，EP01 Crossy Road 已跑通全流程）

---

## 一、环境要求与安装（一次性，约 10 分钟）

### 依赖清单

| 依赖 | 版本要求 | 用途 |
|---|---|---|
| Node.js | ≥ 20 | 录屏脚本 + Remotion 渲染 |
| pnpm | 任意近期版本 | 装依赖 |
| Python 3 | ≥ 3.9（无第三方包依赖） | 生成 runner / 指标 / 音频 |
| 环境变量 `AIHUBMIX_API_KEY` | — | 网关鉴权，写进 shell profile |

### 安装步骤

```bash
# 1. 拿到代码目录（内部直接拷贝 ~/code/model-showdown）
cd ~/code/model-showdown

# 2. 装录屏依赖（playwright + 浏览器内核）
pnpm install
npx playwright install chromium

# 3. 装视频渲染依赖
cd video && pnpm install && cd ..

# 4. 验证网关连通（应返回 OK）
curl -sS https://aihubmix.com/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $AIHUBMIX_API_KEY" \
  -d '{"model":"gpt-5.6-sol","messages":[{"role":"user","content":"Reply with exactly: OK"}],"max_tokens":16}'
```

### （可选）安装 Claude Code skill：`/showdown` 一句话跑全流程

仓库自带 skill（`.claude/skills/showdown/`），让 Claude Code 自动驾驶整条流水线
（解析模型和题目 → 确认 prompt → 生成+录屏 → 质检 → 渲染 → 出文案）：

- **项目内使用（零安装）**：在本仓库任意子目录打开 Claude Code，直接输入
  `/showdown <题目想法>`，项目级 skill 自动被发现
- **全局安装（任意目录可用）**：
  ```bash
  cp -r .claude/skills/showdown ~/.claude/skills/
  ```
- 用法示例：`/showdown 用 kimi-k3 和 gpt-5.6 比一把 3D 弹球`（可点名模型、
  可给参考图路径；不点名则用默认全阵容）

### 目录结构

```
model-showdown/
├── run_showdown.py      # 多模型并行生成（流式 SSE + 重试）

```

## 二、每期操作流程（7 步）

> 在 Claude Code 里直接 `/showdown ep03: <题目想法>` 可自动走完全流程；以下是手动步骤。

### Step 1 — 出题（人工，5 分钟）

新建 `episodes/epNN/task.md`，英文 prompt。**三条铁律**：

1. 单文件 HTML、无 CDN 无外部资源（保证离线可录）
2. **必须强制 auto-demo 模式**（加载 1 秒后自动开始演示）——这是全自动录屏的命门
3. 结尾固定要求：`Return ONLY the complete HTML file in a single ```html code block.`

选题标准：成品自己会动、视觉冲击强、能区分模型水平（游戏 / 3D 场景 / 物理模拟 / 数据可视化）。

### Step 2+3 — 生成 + 录屏（自动，流水线模式，推荐）

```bash
python3 run_showdown.py --task 提示词.md --record 26            # 期号自增，自动建下一个 epNN
python3 run_showdown.py --task 提示词.md --ref 参考图.png --record 26   # 带参考图（多模态发给所有模型）
python3 run_showdown.py episodes/ep02 --record 26               # 也可显式指定期号目录
```

- 三模型并行生成；**`--record 26` 让每个模型的成品在生成完成的瞬间自动开录**，
  与更慢模型的生成重叠——快模型的录屏时间完全被 K3 的思考时间吸收，等于免费
- 流式 SSE + 30 秒心跳日志（`streaming… N chunks, M content chars`），等待不再是黑盒
- metrics.json **增量落盘**（每完成一个模型写一次），中途杀进程不丢数据
- 产出：`work_<model>/index.html` + `raw_<model>.json` + `metrics.json` + `recordings/*`
- 耗时：GPT-5.6 / Opus 4.8 约 2-3 分钟；**Kimi K3 约 10-35 分钟**（强制满级思考，全流程唯一瓶颈）
- 单独补跑某家：`--models kimi-k3`；手动重录：`node record.mjs episodes/epNN --seconds 26 [--model X]`（多模型并行录，3 个 27 秒完成）

### Step 4 — 人工质检（2 分钟）

看 `recordings/poster_*.png` 确认画面；看 `report_*.json` 的 `consoleErrors` / `frozen` 字段。

### Step 5 — 合成渲染（自动 + 1 处人工，约 5 分钟）

```bash
# 音轨（zen=禅意铃音，arcade=8bit 街机，按题材选）
python3 gen_audio.py video/public/epNN/audio.wav --style zen --seconds 26

cp episodes/epNN/recordings/*.webm video/public/epNN/
python3 gen_props.py episodes/epNN "<英文标题>" "<A vs B vs C>" > video/props-epNN.json
# ★ 人工：把 props 里的 verdict 从 SHIPPED 改成有区分度的短语（BEST LOOKING / SLOWEST / FROZE…）

cd video
npx remotion still ShowdownWide out/check.png --frame=300 --props=props-epNN.json   # 抽帧检查布局
npx remotion render ShowdownWide out/epNN-wide.mp4 --props=props-epNN.json
npx remotion render ShowdownSquare out/epNN-square.mp4 --props=props-epNN.json      # 需要方版时
cp out/epNN-*.mp4 ../episodes/epNN/
```

视频结构：直接进三面板对比画面（20 秒，模型名+绿色成本角标，底部 AIHubMix logo 常驻）→ 结尾 "The bill" 成本/耗时对比条形图（4 秒）。

### Step 6 — 发布文案（人工 + AI 辅助，10 分钟）

仿 `episodes/ep01/post.md` 写三变体：**X**（短、punchline 先行）/ **Reddit**（methodology 详述、干货向、忌广告腔）/ **HN**（Show HN + 首评方法论 + 主动列 limitations）。所有数字必须与 metrics.json 一致。

### Step 7 — 复盘（10 分钟）

写 `episodes/epNN/retro.md`（耗时/成本/卡点/下期改进），新教训回写 `~/.claude/skills/showdown/SKILL.md`。

## 三、已知坑（都踩过了）

| 坑 | 处置 |
|---|---|
| Kimi K3 非流式请求 10-15 分钟必被上游断连 | 已改流式 SSE，字节持续回传就不掐 |
| K3 强制满级思考，32k max_tokens 全被推理吃光（$0.48 学费） | K3 的 max_tokens 必须 100k |
| kimi-k3 高峰期 429/503 频繁 | retries=8 硬等；**禁止用 coding-* 通道替跑**（计费口径和模型身份都不真实） |
| 模型名过长挤爆面板头部 | `gen_props.py` 的 `SHORT` 映射加短名 |
| 系统 Python 3.9 | 脚本禁用 `X \| None` 类型注解 |
| Playwright 报浏览器缺失 | `npx playwright install chromium` |

## 四、单期成本与耗时基线（EP01 实测）

- API 成本：约 $1.3（三模型各一次成功 run）+ 废 run 学费另计
- 人工介入点：出题、质检、verdict、文案 ≈ 30 分钟
- 墙钟时间：顺利时约 35 分钟，其中 K3 生成占大头

## 五、运行时间优化（2026-07-17 已实施 + 实测数据）

| 优化 | 状态 | 实测 |
|---|---|---|
| ① 录屏并行（3 个浏览器同时录） | ✅ 已实施 | 3 模型 **27 秒**（原串行 ~2 分钟） |
| ② 流水线化：`--record N` 谁先生成完就先录谁 | ✅ 已实施 | 快模型录屏被 K3 思考时间完全吸收，净开销 0 |
| ③ 流式心跳日志（30s 一行进度） | ✅ 已实施 | 等待可观测 |
| ④ metrics 增量落盘 | ✅ 已实施 | 中途杀进程不再丢数据 |
| ⑤ 渲染提速（concurrency / webm 预转 mp4） | ❌ 实测无收益，不做 | 基线本来就 **22 秒**（此前误判 3 分钟是首次打包）；concurrency 拉满反慢 4s，mp4 输入仅省 0.6s |
| ⑥ K3 生成本身 | 无解，接受 | 满级思考是模型特性；这就是内容看点 |
| ⑦ 错峰跑（cron 半夜生成，早上只做人工环节） | 📋 量产后启用 | 人工等待归零 |

**优化后关键路径 = K3 生成（10-35 min）+ 渲染 22s**——其余全部被重叠或并行掉了。
GPT/Opus 两家的对比期（不含 K3）从提交到可渲染只要 **~3.5 分钟**。
剩余可压项只在人工环节：verdict 和文案让 LLM 出初稿再人工改（30 min → ~15 min）。
